Architecture

DS-STAR Core Loop — FIP-Insight ALL 8 AGENTS IN V1
Analyzer
Profiles every file → D. Ablation: −18pp accuracy if removed. Critical foundation.
→
Planner
One plan step pₖ per round from (q, D, p₀..pₖ₋₁, rₖ₋₁).
→
Coder
Full script sₖ per round — not a delta patch. Q3 ✓
→
Executor
Docker sandbox. 1 container per task, reused across rounds. Q1 ✓
→
Debugger
2-step: traceback summarizer + fixer. Gets (s, tb, D). Max 3 attempts. Q15 ✓
→
Verifier
LLM-judge: "sufficient" → Finalizer. "insufficient" → Router or Finalizer at cap.
→
Router
"Add Step" or backtrack index l → truncate plan, Planner regenerates pₗ.
→
Finalizer
Runs formatting script in Docker sandbox. Q16 ✓
Round loop: Executor → Debugger (on error) → Verifier → Router (insufficient) → Planner → Coder → Executor ↻ · Default 5–6 rounds · Hard cap 40 · 10 KB stdout cap

Sub-Question Generator
Decomposes open-ended query into {sq₁…sqₙ}. Also handles refinement gap-finding — no separate Evaluator. Q11 ✓
→
DS-STAR × N (inner engine, sequential in V1)
Each sub-question runs the full 8-agent pipeline. Produces evidence set {(sqᵢ, aᵢ, sᵢ)}. Parallel: V2
→
Report Writer
Highest-quality LLM. Synthesises cited report. Every claim references sqᵢ + code sᵢ.
→
Generator (refinement)
Checks coverage. Adds sub-questions if gaps found (K=1 rounds). Early stop: zero new questions.
→

Agent-DASC · FIPSection 03: Technology
CBUAE Intelligence Layer
Nine enhancements wrap the DS-STAR core — five ship in V1, four are deferred to V2 to protect delivery scope and validate the core capability first
V2 items are architecturally accommodated via adapter patterns — deferral is scope discipline, not rework. All five V1 enhancements are confirmed in PRD v1.0.
Ships in V1 ✓
E1 · Query Clarity Agent
V1
Validates query answerability. Maps "high-risk customers" → risk_band='High'. Routes Mode A vs B. Shows clarification pop-up before pipeline starts for ambiguous queries.
E2 · Domain-Aware Analyzer
V1
Extends base Analyzer for CBUAE schemas: fraud loss reports (lfi_id, fraud_type, channel, amount_aed, detection_method), SAR/STR records, KYC/CDD, market surveillance. Richer D descriptions.
E3 · Natural Language Explainer
V1
Post-Finalizer node. Converts DataFrames + numbers into analyst-readable narrative. "ABC Bank card fraud rate is 3.2× the peer median at 0.18% of transaction volume." Non-technical analysts read this directly.
E4 · LLM Provider Adapter
V1
Single config: provider: gemini | ollama | azure_openai. Enables on-premise air-gapped deployment via Ollama. All 8 agents use same interface — zero code changes when switching provider.
E5 · Immutable Audit Log
V1
Append-only store. Every query, plan step, code, result linked by task_id. Forever retention in V1. Admin: full access. Analyst: own queries only. Regulatory examination ready from day one.
Deferred to V2 🔮
E6 · PII Masking
V2
V1 data is analyst-uploaded files — not live production DBs. PII masking designed when live connectivity added in V2, based on actual schemas and CBUAE data governance policy (Topic 7).
E7 · Regulatory Knowledge Injection
V2
Dept-specific domain knowledge in Planner/Coder prompts: AML/CFT law, FATF 40, CPR, penalty frameworks. Deferred — V1 treats all analysts identically (Topic 6, Feature Discussion).
E8 · Supervisory Report Templates
V2
Pre-built dept templates: Fraud Benchmarking, SAR Trend, Consumer Scorecard, Evidence Package. Analyst picks template instead of free-form output spec. Added once core system is proven.
E9 · Live Database Connectivity
V2
Informatica Data Catalog integration. Analyst's org-wide catalog access reflected automatically. Adapter pattern accommodated in V1 architecture — seamless plug-in when CBUAE IT is ready.
Why Defer? — Scope Discipline
V1 goal: prove core analysis capability across all 5 departments. Customisation (Topics 6 & 7) adds complexity without validating the core hypothesis. V2 customises a system analysts already trust and use daily.

❌ Dropped or Changed vs Paper
Separate Evaluator Agent DROPPED
Paper had a separate Evaluator for DS-STAR+ refinement. Confirmed (Q11): Sub-Question Generator handles both decomposition AND gap-finding. No separate node.
20-Round Default CHANGED
Paper default: 20 rounds. Our default: 5–6 rounds. Paper shows hard tasks average 5.6 rounds — Extension UX (+5 rounds per click) handles cases needing more depth.
Debugger Design CHANGED
Paper implies single-step Debugger. Our impl: 2-step (traceback summarizer + code fixer). Receives (s, traceback, D) — data descriptions D critical for fixing semantic errors. (Q15)
Finalizer Design CHANGED
Paper implies text-only formatting in Finalizer. Our impl: Finalizer generates and runs a formatting script in Docker sandbox — same execution model as Coder for consistency. (Q16)
Docker Lifecycle CHANGED
Paper implies new container per round. Our impl: one container per task, reused across all rounds — state persists, overhead reduced significantly. (Q1)
Delta/Patch Scripts DROPPED
Paper is ambiguous. Confirmed (Q3): Coder produces a complete new script sₖ each round, not an incremental delta. Simpler — no patch-merging bugs.
